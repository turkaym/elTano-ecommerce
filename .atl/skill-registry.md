## Compact Rules

Pre-digested rules per skill. Delegators copy matching blocks into sub-agent prompts as `## Project Standards (auto-resolved)`.

### branch-pr
- Every PR MUST link an approved issue and include `Closes #N`, `Fixes #N`, or `Resolves #N` in the body.
- Verify the linked issue has `status:approved` before opening a PR.
- Branch names MUST match `^(feat|fix|chore|docs|style|refactor|perf|test|build|ci|revert)/[a-z0-9._-]+$`.
- Every PR MUST have exactly one `type:*` label matching the PR template checkbox.
- PR body must include summary, changed-files table, test plan, and all contributor checklist items.
- Commit messages MUST follow Conventional Commits: `type(scope): description` with allowed types only.
- Do not proceed if automated issue reference, issue approval, type-label, or shellcheck checks fail.

### go-testing
- Prefer table-driven tests for multiple cases; use `t.Run(tt.name, ...)` for each case.
- Test pure functions directly; mock side effects behind interfaces; cover both success and error paths.
- For Bubbletea, test `Model.Update()` state transitions directly before using full interaction tests.
- Use Charmbracelet `teatest.NewTestModel` for end-to-end TUI interaction flows.
- Use golden files for stable visual output and update them only for intentional output changes.
- Use `t.TempDir()` for filesystem tests and skip real-command integration tests under `testing.Short()`.

### issue-creation
- Blank issues are disabled; create issues only from bug report or feature request templates.
- Search existing issues first and ensure all required template fields/pre-flight checkboxes are complete.
- Newly created issues start with `status:needs-review`; a maintainer MUST add `status:approved` before any PR.
- Questions belong in GitHub Discussions, not Issues.
- Bug reports need clear repro steps, expected/actual behavior, OS, agent/client, shell, and relevant logs when available.
- Feature requests need problem description, proposed solution, and affected area.
- Use `gh issue edit <number> --add-label "status:approved"` only for maintainer approval.

### judgment-day
- Before launching judges, resolve compact rules from Engram `skill-registry` or fallback to `.atl/skill-registry.md`.
- Launch exactly two independent blind judge sub-agents in parallel with identical target and standards.
- Judges must classify findings as CRITICAL, WARNING (real), WARNING (theoretical), or SUGGESTION.
- Treat only findings reported by both judges as confirmed; single-judge findings are suspect and need triage.
- Round 1 requires user confirmation before fixing confirmed issues.
- Fix confirmed CRITICALs/real WARNINGs only; do not refactor beyond the confirmed issue list.
- Re-judge after fixes and escalate/ask user after two unresolved fix iterations.

### skill-creator
- Create a skill only for reusable AI guidance, project-specific conventions, complex workflows, or decision trees.
- Do not create a skill for trivial, one-off, or already-documented patterns; reference existing docs instead.
- Skill directories use `skills/{skill-name}/` with required `SKILL.md`, optional `assets/`, optional `references/`.
- `SKILL.md` frontmatter must include `name`, `description` with Trigger text, `license: Apache-2.0`, and metadata author/version.
- Keep critical patterns first; use tables for decision trees; keep code examples minimal.
- Do not add Keywords sections, lengthy explanations, or web URLs in references; use local paths.
- After creating a skill, register it in `AGENTS.md` and verify the checklist.
